# Prototype Instructions

Run the local server yourself and open the preview in the in-app browser. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Portfolio direction

- Build with Astro as a static multilingual site.
- Preserve the approved A+C direction: editorial typography with strategic proof and clear section numbering.
- Use warm paper backgrounds, black ink, restrained accent colors, square corners, and real project imagery.
- The primary conversion is a WhatsApp conversation with `+7 708 639 15 24`.
- Never promise measurable awareness growth. Promise the strategic and visual system required to develop it.
- Main content languages are Russian and Kazakh. Keep meanings equivalent while writing naturally in each language.
- Treat folders ending in `_case` as the authoritative final source. Never use a matching `_teren` folder when `_case` materials are available. Keep `ASSET_SOURCES.md` current.
- Exception recorded by the user: use `Biohumus_teren/brandbook_biohumus.pdf` as the authoritative Russian layout reference for the Biohumus case.
